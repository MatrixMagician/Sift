<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| counter names and labels â†’ CSV cells | Customer-supplied text reaches a file a spreadsheet will open and may evaluate |
| counter names and messages â†’ Markdown cells | Customer-supplied text reaches a document the operator reads in a terminal or viewer |
| analysis models â†’ JSON report | Non-ASCII code points in customer text would otherwise reach the JSON artefact raw |
| `path` argument â†’ filesystem write | The renderer writes wherever it is told; containment is the caller's responsibility |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

The `write_attribution_csv` quoting-suffices argument rests on MCM keys being structurally hex or `[\w:]+`, which does not transfer to header-derived counter names. Verified by `test_csv_formula_guard` across all six triggers. |
| T-13-MDESC | Tampering | `_counter_table` / `_hazard_table` | medium | mitigate | Every dynamic Markdown cell passes through `render.markdown._field`, which wraps the shared `sanitise` plus Markdown and HTML escaping. | T-13-JSONESC | Tampering | `render_perfmon_json` | medium | mitigate | `ensure_ascii=True` backslash-escapes every non-ASCII code point, so the JSON artefact carries no raw C1 or Cf terminal-injection byte. Verified by `test_json_is_pure_ascii`. |

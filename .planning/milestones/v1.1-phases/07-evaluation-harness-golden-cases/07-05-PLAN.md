## Phase Goal

**As a** Sift maintainer, **I want to** run `sift eval` over a committed golden-case suite that scores hypothesis quality and fails when scores regress below `eval/thresholds.toml`, **so that** hypothesis quality is measurable and regression-gated rather than vibes-based.

This plan adds the **optional LLM-as-judge** slice (EVAL-04, ROADMAP SC4): a second, advisory opinion on hypothesis quality reported alongside — never instead of, never gating — the keyword scores.

<objective>
Add opt-in `--judge` grading that uses the SAME local model (via the existing `InferenceClient.chat`) with a versioned `judge.md` prompt to score hypothesis-vs-truth match, reported alongside keyword scores and never affecting the exit code (D-08, EVAL-04).

Purpose: satisfy EVAL-04 / ROADMAP SC4 — a local judge score alongside keyword scores.
Output: judge prompt template + judge module + `--judge` flag + offline advisory test + live judge test.
</objective>

## No new external API (COVERAGE.md opt-out)

This phase integrates NO new external API. `--judge` reuses the already-integrated chat-completions surface of the hand-rolled `InferenceClient` (integrated in Phases 3–4, SSRF-guarded). Therefore no COVERAGE.md capability matrix is added for a new service (one-line opt-out justification per project convention): the only inference surface (chat completions) is already fully integrated — there is no new API to enumerate.

HDBSCAN (L2-normalised, min_cluster_size=2) or the config-driven agglomerative fallback, turn noise

prompt template — persisting vectors, clusters and eager labels through the store.

the prompt to be a file editable without touching Python.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| log content -> label LLM call | Exemplar text (from ingested logs) flows into the label prompt — prompt-injection surface |
| LLM label response -> clusters.label -> terminal | Model-generated label text is untrusted and later rendered |
| prompt template file -> runtime | The versioned template is loaded and interpolated at runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-16 | Tampering | prompt-injection from log content into the label call | medium | mitigate | exemplars only (no instructions echoed as commands); labels are freeform + length-capped + sanitised at render; a hostile label cannot alter control flow (no tool-calling, no schema execution) |
| T-03-17 | Spoofing | adversarial label text with control bytes rendered to the user | medium | mitigate | labels capped by code points; downstream _sanitise (Plan 06 show clusters) strips control/bidi bytes (WR-01 precedent) |
| T-03-18 | Denial of Service | oversized label / prompt blows the context budget | medium | mitigate | PromptBudget breadth-first truncation; per-label code-point cap; one batched call |
| T-03-19 | Integrity | label-parse failure corrupts cluster state | low | mitigate | lenient parse; on failure labels stay NULL and clusters degrade to signature — never crash |
</threat_model>

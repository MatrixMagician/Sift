<objective>
Close gap G1 (high, RAG-03): a reasoning/empty/"no choices" 200 inference response
crashes `sift analyze` with an UNCAUGHT `ValueError("chat response has no choices")`
traceback (exit 1), violating the load-bearing never-crash invariant.

Root cause: `hypothesise()` wraps the generation state machine in `except httpx.HTTPError`
only (hypothesise.py:304). `client.chat()` raises a plain `ValueError` — not an
`httpx.HTTPError` — on a malformed-but-200 body (no `choices`, or non-string/absent
`content`), so it escapes the Outcome machinery and the CLI handler into a raw traceback.

Fix (root-cause, one guard at the shared boundary): (1) broaden the generation-call
handling in `hypothesise()` to catch `ValueError` alongside `httpx.HTTPError` and map it to
a clean `failed` Outcome (nothing persisted, exit 1); (2) normalise empty/whitespace-only
content to the same malformed-response `ValueError` inside `client.chat()`, so the reasoning
model that exhausts its budget on `reasoning_content` (empty `content`, `finish_reason:
"length"`) routes through the identical clean path instead of wasting a doomed repair
round-trip and landing on a different exit code.

Purpose: honour the "LLM output must degrade gracefully — never crash" invariant for the
one un-covered malformed-response shape found in live UAT (Qwen3.5-27B reasoning model).
Output: two guarded source paths + three regression tests, no new dependencies, no CLI change.
</objective>

<failed_vs_degraded_decision>
The malformed/empty 200 body is mapped to `failed` (exit 1), NOT `degraded` (exit 3).

Rationale: `degraded` (exit 3) exists for a run that PRODUCED inspectable output that was
then rejected — malformed JSON or bad citations — where the raw text is persisted as
`triage_raw` so an operator can see what the model returned. A no-choices / absent-content /
empty-content response produced NOTHING inspectable (there is no content string worth
persisting). That is the same situation as a transport failure that produced no output, which
the existing code already maps to `failed`/exit 1 ("nothing new persisted"). Mapping this to
`failed` keeps the two "nothing was produced" cases symmetric and honest; mapping it to
`degraded` would persist an empty/useless `triage_raw` and mislead CI scripting. Record this
in the SUMMARY and in a one-line code comment on the broadened except.
</failed_vs_degraded_decision>

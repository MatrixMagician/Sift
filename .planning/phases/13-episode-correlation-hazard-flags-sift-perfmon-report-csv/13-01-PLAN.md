<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| customer CSV header → adapter `attrs` keys | Counter names are attacker-influenceable text that becomes dictionary keys |
| test fixture bytes → `tmp_path` filesystem | Synthetic fixtures are written under pytest's `tmp_path`, never into the repo tree |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-13-FIXPATH | Tampering | `write_*_csv(tmp_path)` builders | low | mitigate | Every builder writes strictly beneath the pytest-supplied `tmp_path`; no builder accepts a caller-chosen absolute destination, so a fixture can never overwrite repo or user files. |
| T-13-VACUOUS | Repudiation | the three synthetic fixtures | medium | mitigate | Each builder is paired with a test asserting the fixture genuinely carries its property (collision real, widths non-uniform, `float()` accepts while `isfinite` rejects). Without this, a drifted builder would turn downstream security tests green while exercising nothing. |
| T-13-NETWORK | Information Disclosure | new test modules | low | accept | The autouse `_no_network` fixture (`tests/conftest.py:34-54`) already raises on `socket.socket.connect`; nothing in this plan needs network, so no additional control is warranted. |

No package-manager installs occur in this plan (RESEARCH.md § Package Legitimacy Audit: no external
packages are added by this phase), so no `T-13-SC` supply-chain checkpoint applies.
</threat_model>

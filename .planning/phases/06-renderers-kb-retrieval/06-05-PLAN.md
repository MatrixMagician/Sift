<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Markdown/HTML → WeasyPrint | Report content (model/adapter text) is rendered by an HTML engine that can fetch external resources unless blocked |
| pyproject extra → dependency chain | `markdown`/`weasyprint` are third-party packages pulled only for the pdf extra |
| render_pdf → filesystem `--out` | PDF bytes are written to a user-supplied path |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-20 | Info disclosure (egress) | WeasyPrint fetching an external resource | high | mitigate | Rejecting `url_fetcher` blocks ALL fetches + self-contained HTML (inline style, no `<img>`, only `#evt-` links) — egress impossible by content AND by fetcher (D-09) |
| T-06-21 | Tampering | log/model bytes injecting HTML/CSS into the PDF | high | mitigate | Raw text fenced in code blocks (06-01) + `sanitise` on every field; `markdown` output not `unsafe`-flagged; no `<script>` |
| T-06-22 | Info disclosure / DoS | missing extra/pango raising a traceback leaking paths | medium | mitigate | Both `ImportError` and WeasyPrint OSError map to a helpful `PdfExtraMissing` message + exit 1 (D-10, Pitfall 5/7) |
| T-06-SC | Tampering (supply chain) | `markdown` + `weasyprint` in the pdf extra | high | mitigate | Both are PyPI, long-lived, high-reputation (RESEARCH Package Legitimacy Audit: OK/Approved), pinned exactly, and vetted at the 01-01 blocking-human checkpoint + ADR 0002 — no `[ASSUMED]`/`[SUS]`/`[SLOP]` package, so no new legitimacy checkpoint is required; extra is opt-in |
</threat_model>
